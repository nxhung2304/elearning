# Code Review: feature/hung-#10-add-course-category-model → main

## Summary

Implements the `CourseCategory` model with ancestry-based tree, friendly_id slug generation, soft-delete via Discard, and full i18n support. The implementation is well-structured and matches the issue spec closely — the main findings are a logic bug in the parent validation, a slug uniqueness test that doesn't exercise friendly_id's actual behavior, and boilerplate noise in the initializer.

---

## Issues

### Critical

- `app/models/course_category.rb:47` – `parent_must_be_kept` accesses `parent` (a DB query) using `parent_id.present?` as the guard, but ancestry stores `parent_id` as a virtual attribute derived from the `ancestry` string. On an unsaved record `ancestry` is nil, so `parent_id` is nil even when `:with_parent` is used — this guard may never fire on `build`. Use `ancestry.present?` (the actual persisted column) or call `parent_id` only after the record is saved. Safer: `validate :parent_must_be_kept, if: -> { parent.present? }` with the `parent` method from ancestry.

### Warning

- `app/models/course_category.rb:24` – `extend FriendlyId` should come before `friendly_id :name, use: :slugged`, but more importantly the class-level ordering is inconsistent: `include` macros at the top, then `extend`, then `friendly_id`, then `has_ancestry`. Rails convention groups `include`/`extend`/concern macros first, then associations, then validations, then callbacks. Reorder: includes → associations → validations → callbacks → public methods → private.

- `test/models/course_category_test.rb:48-54` – The "slug is unique" test checks `category.errors[:slug]` after `category.valid?`, but friendly_id generates the slug in `before_validation`. When a duplicate name is given, friendly_id appends `-2` suffix automatically — the record will be *valid*, not invalid. This test is asserting the wrong behavior: uniqueness validation fires only if someone manually sets a duplicate `slug`. Test should instead assert that two categories with the same name get *different* slugs (e.g. `"programming"` vs `"programming-2"`).

- `test/factories/course_categories.rb:22` – `name { Faker::Name.name }` uses person names (e.g. "John Doe") for categories. `Faker::Lorem.word` or `Faker::Commerce.department` would better represent category data and is less likely to produce collision issues in parallel test runs with the uniqueness index on `slug`.

### Suggestion

- `config/initializers/friendly_id.rb` – The file is the default template generated by `rails generate friendly_id` — 107 lines, 90+ of which are commented-out examples and documentation. Trim to only the active configuration (the `config.use :reserved` block and reserved words). Dead comment blocks are noise in a checked-in initializer.

- `db/migrate/20260531073121_create_friendly_id_slugs.rb:1-6` – The `MIGRATION_CLASS` conditional is a compatibility shim for ActiveRecord < 5, which this project cannot be (Rails 8.1). Remove the conditional and use `ActiveRecord::Migration[8.1]` directly, matching the style of the sibling migration.

- `app/models/course_category.rb:28` – `validates :slug, presence: true, uniqueness: true` — the `uniqueness: true` is redundant given the DB unique index and friendly_id's suffix mechanism. It also causes an extra `SELECT` on every save. The DB constraint is the real guard; the model-layer uniqueness check only helps surface a friendly error message, but friendly_id handles uniqueness internally. If keeping it for UX, add a comment explaining why.
